# Changelog

All notable changes to Featly will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

Until version `1.0.0`, the public API is unstable and minor versions may introduce breaking changes. Breaking changes will be called out explicitly in the notes for each release.

## [Unreleased]

### Added

- **Statistical significance for experiment analytics.** `GET /api/admin/experiments/{key}/analytics` gains a per-metric two-proportion z-test (the standard large-sample equivalent of a 2x2 chi-square test) comparing each variant against a baseline — the flag's default variant, falling back to the first-observed variant when the default was never exposed. Each `MetricAnalytics` now carries `pValue`, `isSignificant` (the conventional `p < 0.05` threshold), and `upliftVsBaseline` (relative rate change); a new `winners` list names, per metric, the highest-converting variant among those significantly *better* than the baseline (a significant *drop* never wins). Implemented in-house (`SignificanceCalculator`, no statistics package — matching the project's `SemverComparer`/`MurmurHash3` precedent) as a pure, additional pass over the existing `ExperimentAnalyticsAggregator`; no schema or event-shape change. The Experiments dashboard shows a baseline badge, a `(p=…, significant)` annotation per variant bar, and a "Winner: …" callout per metric. This is a **fixed-horizon** test — repeatedly peeking at a running experiment inflates the false-positive rate; sequential analysis remains on the post-1.0 roadmap. Closes "Statistical significance for experiments" from `PLAN.md`'s post-1.0 list.
- **API-key expiry + rotation.** `ApiKey` gains an optional `ExpiresAt` (UTC): an expired key stops authenticating on the next request while its row stays for audit, and the mint endpoint/dashboard/CLI accept the new expiry (`featly apikey generate --expires-in <days>`). New `POST /api/admin/apikeys/{id}/rotate` (gated by `ApiKeyCreate` + `ApiKeyRevoke`) mints a replacement carrying the old key's name, scope, environment, and user binding, then revokes the old key — the replacement token is returned exactly once; `featly apikey rotate <id>` drives it from the CLI and the dashboard API-keys screen gets a Rotate action plus an "expired" status badge. Publishes a new `apikey.rotated` domain event (audit + webhooks). Migration `AddApiKeyExpiry`. Closes the "API-key rotation / expiry" security follow-up from the v0.1.0 audit.
- **Opt-in request rate limiting.** New `Featly:RateLimiting` section (bound into `FeatlyRateLimitOptions`, DB-overridable via `GET/PUT /api/admin/settings/rate-limit` and the dashboard Settings screen — "DB beats config"): fixed one-minute windows partitioned per surface (`/api/auth/*` for login brute-force protection, `/api/admin/*`, `/api/sdk/*`) and per client (authenticated identity when present, else remote IP). Off by default; `0` means unlimited for a surface; rejections return `429` with `Retry-After: 60`. Implemented as a single endpoint filter at the `/api` group root, so the embedded quickstart stays "two DI calls + a mount" with no host pipeline change. Closes the "request rate limiting" security follow-up from the v0.1.0 audit.
- **Synchronizer-token CSRF layer for the dashboard session.** Login now mints a random per-session anti-forgery token, stored as a claim inside the `HttpOnly` auth cookie and echoed in the login/`GET /api/auth/me` JSON (`csrfToken`); every cookie-authenticated mutation must present it in the `X-Featly-Csrf` header (constant-time compare, enforced by a single filter at the `/api` group root). Bearer requests are exempt — a header credential is not ambiently attached by browsers — and the dashboard sends the header automatically. Sessions minted before the upgrade fail closed with a hint to sign in again. Closes the "synchronizer-token CSRF layer" follow-up from the v0.1.0 audit — all four audit follow-ups are now shipped.
- **Dedicated `BackupExport` / `BackupImport` permissions.** `GET /api/admin/export` and `POST /api/admin/import` no longer piggyback on `FlagRead`/`FlagCreate` — a bundle spans flags, configs, and segments, so the coarse gate let a flag-only role move entity kinds it could not touch individually (the SECURITY_AUDIT.md follow-up). The two new permissions are held only by the system Admin role by default (the boot-time seed refreshes system roles, so existing installs pick them up automatically); grant them to a custom role for backup tooling. **Behavior change (pre-1.0):** a Viewer can no longer export and an Editor can no longer import — use an admin key or a custom role.

## [0.0.8-preview.1] - 2026-05-29

Ships M10 end-to-end: outbound webhooks + a richer audit log. A shared internal domain-event publisher fans every consequential action — flag/config/segment/experiment mutations, M8 approval decisions, and M7 RBAC changes — out to two consumers. The first persists each as an immutable `AuditEntry`, queryable through `GET /api/admin/audit` with entity/actor/date filters. The second is the webhook dispatcher: registered `WebhookEndpoint`s subscribe to event types (or all) and an optional environment, and a background worker drains a persisted, restart-surviving delivery queue — signing each body with the endpoint's secret (`X-Featly-Signature: sha256=…`, HMAC-SHA256) and retrying with exponential backoff until success or dead-letter. The embedded dashboard grows Webhooks management (list/create/edit/test/delete + delivery status) and an Audit log screen. Everything is additive and opt-in — no breaking change.

### Added

- **M10 PR 10D — Dashboard Webhooks management + Audit log (closes M10).** The webhook + audit features become operable from the embedded dashboard. New **Webhooks** screen lists endpoints (name, URL, enabled, subscribed events) with an inline create form (name, URL, optional comma-separated event types, optional secret — blank auto-generates); the detail view edits the endpoint (name/URL/enabled/event-types/secret), **sends a test event**, deletes it, and shows a recent-deliveries table (event, status badge, attempts, last status code, last error). New **Audit log** screen queries `GET /api/admin/audit` with entity-type and actor filters, rendering when / action / entity / actor newest-first. Pure front-end (`app.js` / `app.css` / `index.html`); no API changes. **M10 is complete.** No breaking change.
- **M10 PR 10C — Webhook delivery engine: admin CRUD + dispatcher + retry worker.** Brings outbound webhooks online (ARCHITECTURE.md §17). New `/api/admin/webhooks` CRUD behind the admin policy with per-route `Webhook*` permissions (`POST` auto-generates a signing secret when none is given; `GET /{id}/deliveries` lists recent attempts; `POST /{id}/test` enqueues a synthetic `webhook.test` delivery). A second event consumer — `WebhookDispatcher` — joins the 10B pipeline: for each enabled endpoint whose subscription matches the event type (empty = all) and environment filter, it enqueues a `WebhookDelivery` carrying the serialized payload envelope. The `WebhookDeliveryWorker` background service drains the persisted queue on an interval, signs each body with the endpoint's secret (`X-Featly-Signature: sha256=…`, HMAC-SHA256, plus `X-Featly-Event`/`X-Featly-Delivery` headers), POSTs it, and writes the outcome back — a 2xx marks the row `Succeeded`; otherwise it reschedules with exponential backoff (base·2^(n-1), capped) until the attempt budget is spent and the row is dead-lettered. Worker tuning binds from `Featly:Webhooks`. 7 new tests (HMAC matches a manual computation + varies by secret; dispatcher enabled/type/env matching; backoff growth + cap; webhook CRUD; test-enqueue; flag-update dispatches only to matching endpoints; SDK-scope rejection). 292 tests passing total. No breaking change.
- **M10 PR 10B — Domain-event backbone + audit recorder + audit query API.** Wires up the shared event pipeline (ARCHITECTURE.md §17). New server-internal `IFeatlyEventPublisher` fans a `FeatlyDomainEvent` out to every registered `IFeatlyEventConsumer`, isolating failures so a misbehaving consumer never breaks the mutation that produced the event. The first consumer, `AuditRecorder`, persists each event as an `AuditEntry`. Every admin mutation now publishes after it succeeds: flags (created/updated), configs (created/updated), segments (created/updated/deleted), experiments (created/updated/started/stopped), approval changes (approved/rejected/applied, incl. emergency bypass), and RBAC role assignments (assigned/unassigned) — the "everything" audit scope, carrying the actor, environment, entity key, and an `after` payload. New `GET /api/admin/audit` (`AuditRead`) returns entries newest-first with optional `entityType` / `entityKey` / `actor` / `env` / `from` / `to` / `limit` filters. The webhook dispatcher joins as a second consumer in 10C. 4 new server tests (publisher failure-isolation through DI, create→update audit trail, entity-type filter, SDK-scope rejection). 285 tests passing total. No breaking change.
- **M10 PR 10A — Webhooks + audit domain + storage.** First slice of M10 (ARCHITECTURE.md §17). Four new types in `Featly.Abstractions`: `WebhookEndpoint` (url, per-endpoint HMAC `Secret`, enabled flag, a subscribed `EventTypes` list — empty means all — and an optional environment filter), `WebhookDelivery` (the persisted retry queue row — `Status` Pending/Succeeded/Dead, `AttemptCount`, `NextAttemptAt`, last status/error, with the exact signed `Payload` captured at enqueue), `AuditEntry` (immutable log row — action, entity type/key, environment, actor, and a `JsonElement` `Data` for before/after), and `FeatlyDomainEvent` (the in-process event the 10B publisher fans out to the audit recorder and the webhook dispatcher) plus a `FeatlyEventTypes` constants class (`flag.updated`, `change.approved`, `role.assigned`, …). Three sub-stores on the storage facade — `IWebhookStore` (CRUD), `IWebhookDeliveryStore` (enqueue, due-claim, result update, by-endpoint listing) and `IAuditStore` (append + filtered query) — with InMemory + SQLite providers and EF migration `AddWebhooksAndAudit` (event-types via `PrimitiveCollection`, the audit `Data` as a JSON column, delivery status as its enum name, ticks for all timestamps, and indexes on `(Status, NextAttemptAt)` for the worker claim plus audit `At`/`(EntityType, EntityKey)`/environment). No endpoints or delivery engine yet — 10B wires the event backbone + audit API, 10C the webhook CRUD + dispatcher + delivery worker, 10D the dashboard. 3 new SQLite round-trips (webhook event-types + delete, delivery due-claim + result, audit payload + filters). 281 tests passing total. No breaking change.

## [0.0.7-preview.1] - 2026-05-28

Ships M9 end-to-end: experiments / A-B testing. An `Experiment` is layered on an existing flag (by key) with a set of conversion metric keys, an optional sticky-assignment toggle, and a start/stop window. While an experiment is active the SDK automatically emits an `Exposure` event whenever it evaluates the underlying flag, and applications track conversions via `IFeatlyClient.Events.TrackAsync`; events batch through a non-blocking channel and flush to `POST /api/sdk/events`. The server aggregates the raw event rows on read into per-variant exposure counts and per-metric conversion rates (`GET /api/admin/experiments/{key}/analytics`), and the embedded dashboard grows an Experiments screen with a list, a detail view, Start/Stop controls, and CSS-only bar charts. Sticky experiments pin a subject to the first variant it saw so a mid-flight weight change doesn't migrate already-exposed subjects. Everything is additive and opt-in — no breaking change.

### Added

- **M9 PR 9D — Dashboard Experiments list + detail + analytics charts (closes M9).** The experimentation feature becomes operable from the embedded dashboard. New **Experiments** nav screen lists the environment's experiments (key, name, underlying flag, running/stopped/draft status, metric count, started date). The **detail view** shows the experiment's metadata (name, hypothesis, flag link, sticky flag, metric keys, window), a **Start / Stop / Restart** control wired to the 9B lifecycle endpoints, and an **analytics** panel rendered from `GET /api/admin/experiments/{key}/analytics`: an "exposures by variant" bar chart plus one conversion-rate bar chart per metric key (per-variant rate with the converting/exposed counts), with an empty-state when no exposures have been recorded yet. Pure front-end (`app.js` / `app.css` / `index.html`); no API changes. Bar charts are CSS-only (no chart library) to keep the dashboard dependency-free. **M9 is complete.** No breaking change.
